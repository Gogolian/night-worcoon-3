Configs

TL;DR One JSON file per "profile" lives in configs/. Each file tells the proxy what port to listen on, what upstream to forward to, and which plugins to run. The TUI lists every .json in that folder.

Minimal example

{
  "name": "httpbin",
  "port": 8080,
  "target": "https://httpbin.org",
  "changeOrigin": true,
  "plugins": ["mock", "recorder"],
  "storage": { "type": "fs", "path": "./recordings/httpbin" }
}

Top-level fields

FieldTypeRequiredWhat it does
namestringyes Human label shown in the TUI and used as the default storage subfolder.
portnumberyes Local port the proxy listens on.
targetstring (URL)yes Upstream base URL, e.g. https://httpbin.org.
changeOriginbooleanno Rewrites the outgoing Host header to match the target. Almost always true for HTTPS upstreams.
followRedirectsbooleanno If true, the manual forwarder follows 301/302/303/307/308 internally. Default: false.
securebooleanno Set false to skip TLS certificate verification when forwarding. Default: true.
requestHeadersobjectno Headers added/overridden on every outgoing request to the upstream.
pluginsstring[]no Names of plugins to enable, in pipeline order. Only plugins listed here are loaded for this profile.
storageobjectno Recording backend. See Storage.
mockobjectno Config for the mock plugin. See Plugins → mock.
recorderobjectno Config for the recorder plugin.
bucketobjectno Config for the bucket plugin.
corsobjectno Config for the cors plugin.
latencyobjectno Config for the latency plugin.
wsTapobjectno Config for the ws-tap plugin.
Plugin order matters plugins is the pipeline order. The recommended ordering is ["cors", "latency", "bucket", "mock", "recorder"] — that way cors handles preflights first, latency delays/fails before any work is done, bucket handles CRUD, mock handles canned/replayed responses, and recorder only sees what actually went upstream.

Storage shorthand

"storage": { "type": "fs",     "path": "./recordings/httpbin" }
// or
"storage": { "type": "sqlite", "path": "./recordings/httpbin.db" }

If you omit storage, the proxy defaults to fs at ./recordings/<name>.

Plugin sub-blocks (cheat sheet)

Each plugin reads its own top-level key. Full reference is on the Plugins page.

mock

"mock": {
  "rules": [
    { "method": "GET", "url": "/users/:id", "action": "RET_REC", "fallback": "PASS" },
    { "method": "POST", "url": "/login",    "action": "MOCK",
      "response": { "status": 200, "body": { "token": "abc" } } },
    { "method": "*", "urlContains": "/debug", "action": "PASS" }
  ]
}

Actions: PASS · MOCK · RET_REC. Fallbacks (RET_REC miss): "PASS", "empty200", or "500" (default).

recorder

"recorder": { "recordAll": false }

recordAll: true also captures mocked / replayed responses.

bucket

"bucket": {
  "persistPath": "./recordings/myproxy-bucket.json",
  "collections": [
    { "path": "/api/users",    "idPattern": "numeric" },
    { "path": "/api/sessions", "idPattern": "uuid",
      "responseTemplate": { "session": "{{id}}", "user": "{{user}}" } },
    { "path": "/api/tokens",   "idPattern": "regex:[A-Z]{4}-\\d{4}" }
  ]
}

idPattern: uuid · numeric · alphanumeric · regex:<pattern>.

cors

"cors": {
  "origin": "*",                                     // or "https://app.example.com" or ["a","b"]
  "methods": "GET,POST,PUT,PATCH,DELETE,OPTIONS",
  "headers": "Content-Type,Authorization",
  "exposeHeaders": "X-Request-Id",
  "credentials": false,
  "maxAge": 600
}

latency

"latency": {
  "delayMs": 300,
  "jitterMs": 200,
  "failRate": 0.1,
  "failStatus": 503,
  "failBody": "upstream down",
  "rules": [
    { "urlContains": "/slow", "delayMs": 3000 },
    { "method": "POST", "url": "/api/flaky", "failRate": 0.5 }
  ]
}

wsTap

"wsTap": {
  "log": true,
  "record": true,
  "recordPath": "./recordings/myproxy-ws",
  "rules": [
    { "url": "/socket", "direction": "client", "textContains": "ping",
      "action": "MOCK", "response": "pong" },
    { "url": "/socket", "direction": "client", "action": "REPLAY",
      "fallback": "PASS" }
  ]
}

Actions: PASS · DROP · MOCK · REPLAY/RET_REC. direction is "client" or "server".

Rule matching (used by mock + latency + ws-tap)

FieldBehaviour
urlContains: "/api"Substring match against the path.
url: "/users/:id"Regex with :segment placeholders for one path component.
url: "/exact/path"Exact equality.
method: "GET"Restrict to a verb (compared upper-case).
method: "*"Any verb. Same as omitting it.

Implementation: src/match.js — compileRule() decides between substring, regex, exact, or any.

Worked examples shipped with the repo

  • configs/httpbin.json — mock + recorder against httpbin.
  • configs/example-bucket.json — bucket + mock + recorder, three collections, three id patterns.
  • configs/example-sqlite.json — same idea but with SQLite storage.
