Plugins

TL;DR Plugins are JS files in plugins/ that hook into the request pipeline. Each profile picks which ones to enable in its plugins array. onRequest stops at the first plugin that sets ctx.response; onResponse always runs everything.

How plugins are loaded

Implementation: src/plugins.js · loadPlugins().

  1. List every .js file in plugins/.
  2. import() each one and grab its default export. The export is either a plain object or a factory function ({config, logger}) => plugin.
  3. Discard any plugin whose name isn't in config.plugins.
  4. Call plugin.init({config, logger}) if defined.
  5. Sort the kept plugins by their position in config.plugins. That order is the pipeline order.

The pipeline

Implementation: runOnRequest / runOnResponse in src/plugins.js, called from src/proxy.js.

// onRequest: stops at first short-circuit
for (const p of plugins) {
  if (ctx.response) break;
  if (p.onRequest) await p.onRequest(ctx);
}

// onResponse: always runs every plugin
for (const p of plugins) {
  if (p.onResponse) await p.onResponse(ctx);
}
Mental model onRequest = "do I want to answer this?"
onResponse = "now that we have an answer, do I want to watch / decorate it?"

cors

Source: plugins/cors.js

Adds Access-Control-* headers to every response and short-circuits OPTIONS preflights with 204.

"cors": {
  "origin": "*",                                     // string | string[]
  "methods": "GET,HEAD,POST,PUT,PATCH,DELETE,OPTIONS",
  "headers": "Content-Type,Authorization",
  "exposeHeaders": "X-Request-Id",                   // optional
  "credentials": false,
  "maxAge": 600
}
  • origin: "*" + credentials: false → echoes *.
  • origin: "*" + credentials: true → echoes the request Origin (because * is illegal with credentials).
  • origin as an array → echoes the request origin if it's allowed; otherwise the first entry.
  • Any non-* or credentialed setup adds Vary: Origin.

latency

Source: plugins/latency.js

Adds artificial delay and an optional random failure rate. Per-route rules override the global defaults; the first matching rule wins.

"latency": {
  "delayMs": 500,                // base delay
  "jitterMs": 200,               // + uniform random [0, jitter)
  "failRate": 0.1,               // 0..1
  "failStatus": 503,
  "failBody": "upstream down",   // string → text/plain, object → application/json
  "rules": [
    { "urlContains": "/slow", "delayMs": 3000 },
    { "method": "POST", "url": "/api/flaky", "failRate": 0.5 }
  ]
}
Where the delay applies Delay runs in onRequest, which means it happens before mock/bucket can short-circuit. So mocked and bucketed responses get throttled too. Failure injection short-circuits with a synthetic response and sets ctx.meta.source = 'latency_fail'.

bucket

Source: plugins/bucket.js

A built-in mock data store with REST-ish CRUD semantics. Each "collection" is a configured base path with an id pattern. Misses are non-blocking — unmatched paths fall through to the next plugin.

"bucket": {
  "persistPath": "./recordings/myproxy-bucket.json",  // omit for in-memory only
  "collections": [
    {
      "path": "/api/users",
      "idPattern": "numeric"
    },
    {
      "path": "/api/sessions",
      "idPattern": "uuid",
      "responseTemplate": { "session": "{{id}}", "user": "{{user}}" }
    },
    {
      "path": "/api/tokens",
      "idPattern": "regex:[A-Z]{4}-\\d{4}"
    }
  ]
}

Routes per collection

MethodPathEffect
POST /coll Create a resource. Auto-generate id, or honor body.id if it matches the pattern. 201.
GET /coll List all resources. 200.
GET /coll/:id Fetch one. 200, miss → fall through.
PATCH /coll/:id Shallow merge. 200, miss → fall through.
PUT /coll/:id Replace (id preserved). 200, miss → fall through.
DELETE /coll/:id Remove. 204, miss → fall through.

idPattern values

ValueGeneratesValidates
uuid crypto.randomUUID()Standard UUID regex.
numeric Auto-increment counter ^\d+$. Counter is rebuilt from persisted data on startup so ids are never reused.
alphanumeric 8-char base64url-ish (A–Za–z0–9)^[A-Za-z0-9]+$
regex:<pat>Tries 5 random alphanumerics that match; falls back to hex.Your regex.

responseTemplate

If set, the response body is the template with {{field}} placeholders replaced by values from the stored record. Dotted paths work ({{user.name}}). Missing values become empty strings. Non-matching collections always return the raw record.

Persistence

If persistPath is set, the whole state is rewritten to that file (pretty-printed JSON) after every mutation. On startup, state is reloaded and numeric counters are recomputed from the largest existing id.

mock

Source: plugins/mock.js

Walks config.mock.rules in order. The first rule whose method + URL matcher accepts the request decides what happens. If no rule matches, the request continues down the pipeline.

"mock": {
  "rules": [
    { "method": "GET", "url": "/users/:id", "action": "RET_REC", "fallback": "PASS" },
    { "method": "POST", "url": "/login",    "action": "MOCK",
      "response": { "status": 200, "headers": {...}, "body": {...} } },
    { "method": "*", "urlContains": "/debug", "action": "PASS" }
  ]
}

Actions

ActionEffect
PASS Forward to the upstream — same as having no rule.
MOCK Return the inline response. If body is a non-Buffer object, it's JSON.stringify'd and content-type defaults to application/json. Sets ctx.meta.source = 'mock'.
RET_REC Look up a recording with storage.findBest(). Sets ctx.meta.source = 'ret_rec' on hit. On miss, uses fallback (see below).

RET_REC matching

  • No query string on the request → return the newest recording matching method + path.
  • Query string present → require exact method + path + query match.

RET_REC fallback (on miss)

ValueBehaviour
"PASS" Don't short-circuit; let the request go upstream.
"empty200" Return 200 with empty body.
"500" (default)Return 500 with a "no recording" message.

recorder

Source: plugins/recorder.js

Pure observer. Runs in onResponse. By default it only saves responses where ctx.meta.source === 'proxy', i.e. the response actually came from the upstream. Set "recordAll": true to also capture mocked / replayed / bucketed responses.

"recorder": { "recordAll": false }

Each saved entry has the shape documented in Storage → Recording shape.

tap

Source: plugins/tap.js · TUI: plugins/tap.tui.js

Pure observer. Runs in onResponse and emits a transaction event for every completed HTTP request. The Tap TUI tab keeps an in-memory live stream showing method, status, latency, response size, source, and URL. It can filter by URL substring or status, pause/resume rendering, clear the buffer, and copy the selected request as a cURL command.

"plugins": ["mock", "recorder", "tap"]

Status filters accept exact codes (200), classes (2xx), ranges (400-499), and comparisons (>=500).

metrics

Source: plugins/metrics.js · TUI: plugins/metrics.tui.js

Tracks per-route statistics and exposes them via a Prometheus-compatible HTTP endpoint and a live TUI tab.

Place "metrics" first in config.plugins so it observes every response, regardless of which plugin short-circuited the request.

"metrics": {
  "endpoint": "/__metrics",   // optional; default /__metrics
  "maxSamples": 1000          // optional; rolling latency window per route
}

Prometheus endpoint

GET <endpoint> (default GET /__metrics) returns text/plain; version=0.0.4 with the following metrics:

Metric nameTypeDescription
proxy_requests_totalcounterTotal requests per route.
proxy_request_errors_totalcounter5xx responses per route.
proxy_bytes_in_totalcounterRequest body bytes received per route.
proxy_bytes_out_totalcounterResponse body bytes sent per route.
proxy_request_duration_mssummaryLatency percentiles (p50/p95/p99) from the rolling window, plus a _count line.

All metrics carry method and path labels. Requests to the endpoint itself are not counted.

TUI tab — Metrics

When metrics is enabled the Metrics tab appears in the TUI. It shows a scrollable table with one row per distinct METHOD + path combination.

ColumnDescription
REQSRequest count since last reset (or startup).
ERR%5xx error rate as a percentage.
p50 / p95 / p99Latency percentiles (ms) from the plugin's rolling sample window.
IN / OUTBytes received / sent since last reset.

Pressing r resets the displayed counters (delta mode — the plugin's cumulative counters and the Prometheus endpoint are unaffected). Percentiles always reflect the full rolling window.

ws-tap

Source: plugins/ws-tap.js

Runs in onWsMessage for every WebSocket text/binary frame. It can log frames, persist sessions to JSON, drop or mock matching client frames, and replay server frames from the newest recorded session for the same path.

"wsTap": {
  "log": true,
  "record": true,
  "recordPath": "./recordings/myproxy-ws",
  "rules": [
    { "url": "/socket", "direction": "client", "textContains": "ping",
      "action": "MOCK", "response": "pong" },
    { "url": "/socket", "direction": "client", "action": "REPLAY",
      "fallback": "PASS" }
  ]
}

Actions

ActionEffect
PASSForward the frame unchanged.
DROPSuppress the frame and forward nothing.
MOCKDrop the original by default and inject response to the opposite peer. Set dropOriginal:false to also forward the original.
REPLAY/RET_RECRead the newest recorded session for the path and inject the next frame from the opposite direction. On miss, fallback:"PASS" forwards normally or "DROP" suppresses the frame.

Rule matching

Rules use the same url / urlContains matching helpers as HTTP mocks, plus optional direction ("client" or "server"), textContains, and textEquals.

har-export

Source: plugins/har-export.js · TUI: plugins/har-export.tui.js

Pure observer. Runs in onResponse and accumulates every completed HTTP transaction as a HAR 1.2 entry. Captures can be downloaded as a .har file via a TUI key or an HTTP control endpoint, then opened in Chrome DevTools, Charles, Fiddler, or Postman for timeline analysis.

"plugins": ["tap", "har-export"],
"harExport": {
  "outputPath": "./recordings/capture.har",   // default save path
  "captureAll": false,                        // false = proxy-source only
  "controlPath": "/_har"                      // HTTP control prefix
}

HTTP control endpoints

The plugin intercepts a small set of paths on the proxy's own port (controlPath prefix, default /_har) so you can automate exports from scripts or CI without a TUI session.

MethodPathEffect
GET /_har/export Return the current capture as application/json (HAR 1.2). Includes full request + response bodies.
POST /_har/save Write the capture to outputPath on disk. Returns { ok, path, entries }.
POST /_har/reset Clear the in-memory capture buffer. Returns { ok, cleared }.

TUI tab (HAR)

When har-export is enabled the TUI gains a HAR tab that shows a live feed of captured entries and the path of the most recent export.

KeyAction
ePrompt for a file path (default = outputPath) and write a HAR summary.
xClear the TUI's local entry list.
pPause / resume live capture in the TUI.
↑ / ↓Scroll the entry list.
Two capture contexts The plugin's HTTP control endpoints and the TUI tab maintain independent in-memory buffers. The HTTP buffer holds full request/response bodies (suitable for Postman import); the TUI buffer holds summary rows (suitable for timeline views). Use the TUI e key or POST /_har/save depending on what level of detail you need.
