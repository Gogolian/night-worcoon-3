Write a plugin

TL;DR Drop a file in plugins/ that export defaults either an object or a factory. Set name, optionally implement onRequest / onResponse / init, then add the name to your config's plugins array.

Skeleton

Save as plugins/my-plugin.js:

// Factory form — recommended. Gets `{ config, logger }` at load time.
export default function create({ config, logger }) {
  // Read your plugin's slice of the profile JSON, e.g. config.myPlugin
  const settings = config.myPlugin || {};

  return {
    name: 'my-plugin',                          // must match the entry in config.plugins

    async init({ config, logger }) {
      // optional — runs once at startup
    },

    async onRequest(ctx) {
      // Inspect ctx.req. To short-circuit, set ctx.response and (optionally)
      // ctx.meta.source. The pipeline stops at the first plugin that does this.
      //
      // Example: answer "/ping" with "pong"
      if (ctx.req.path === '/ping') {
        ctx.response = {
          status: 200,
          headers: { 'content-type': 'text/plain' },
          body: Buffer.from('pong'),
        };
        ctx.meta.source = 'my-plugin';
      }
    },

    async onResponse(ctx) {
      // Always runs (for every plugin). Inspect or mutate ctx.response.
      // Example: stamp a header onto every reply.
      if (ctx.response && ctx.response.headers) {
        ctx.response.headers['x-served-by'] = 'my-plugin';
      }
    },
  };
}

Now activate it in your profile:

{
  "name": "demo",
  "port": 8080,
  "target": "https://httpbin.org",
  "plugins": ["my-plugin"]
}

Object form (no config dependency)

// Equally valid for trivial plugins:
export default {
  name: 'logger-shim',
  async onResponse(ctx) {
    ctx.logger.info(`served ${ctx.req.method} ${ctx.req.url} → ${ctx.response?.status}`);
  },
};

The ctx object

Identical for both hooks. Reference:

ctx = {
  config,    // parsed profile JSON
  storage,   // FsStorage | SqliteStorage — has save(rec) and findBest(...)
  logger,    // shared logger (info, warn, error, debug, trace)

  req: {
    method,    // 'GET'
    url,       // raw, may include '?...'
    path,      // '?'-stripped
    query,     // string after '?', '' if none
    headers,   // node http headers
    body,      // Buffer
  },

  response: null | { status, headers, body },   // set to short-circuit
  meta:    { source: null | 'proxy' | 'mock' | 'ret_rec'
                          | 'ret_rec_fallback' | 'bucket' | 'cors'
                          | 'latency_fail' | <your tag> },
}

The two hooks

onRequest(ctx)

  • Runs before the upstream is contacted.
  • The pipeline stops at the first plugin that sets ctx.response to a truthy value.
  • If you don't want to short-circuit, just leave ctx.response alone.

onResponse(ctx)

  • Runs after a response exists — whether it came from a previous plugin, the upstream, or the manual forwarder.
  • Every enabled plugin's onResponse runs, in pipeline order. There is no short-circuit here.
  • Look at ctx.meta.source to decide whether you care (e.g. recorder skips non-'proxy' sources by default).
  • Mutate ctx.response.headers / body / status in place — your changes are what the client sees.

Patterns from built-in plugins

PatternWhere it's used
Match by URL rule (substring / :id regex / exact) plugins/mock.js, plugins/latency.js — both use compileRule / matchRule from src/match.js.
Persist state to disk on every mutation plugins/bucket.js — saveState() writes the whole map to persistPath as JSON.
Save a recording via the storage backend plugins/recorder.js — uses buildRecord() from src/storage.js then ctx.storage.save().
Read a recording for replay plugins/mock.js (action RET_REC) — ctx.storage.findBest({method, path, query}) then decodeBody().
Decorate every response in onResponse plugins/cors.js — merges CORS headers into whatever response is on ctx.

Imports you can rely on

// URL/rule matching helpers
import { compileRule, matchRule, splitPathQuery } from '../src/match.js';

// Recording shape + helpers
import { buildRecord, decodeBody } from '../src/storage.js';

Gotchas

Bodies are Buffers ctx.req.body and ctx.response.body are Buffers, never strings. Use buf.toString('utf8') to read text, and wrap your output in Buffer.from(...) when building a response.
Order matters Your plugin only runs in the position you put it in config.plugins. Putting recorder before mock, for instance, would have the recorder save mocked responses (since it's seeing them in onResponse regardless — but the order still controls who short-circuits first on onRequest).
Plugins discovery is by directory All .js files in plugins/ are import()ed at load time, even if they're not enabled. Don't put helper files there — put them next to your plugin in a subfolder, or in src/.
No WebSocket hooks (yet) ws:// upgrades bypass the plugin pipeline entirely. If you need to intercept WS frames, you'll have to extend src/proxy.js directly.

Testing your plugin

There's no test framework in the repo. The fastest loop is:

  1. Add your plugin to plugins/.
  2. Reference it in a config under configs/.
  3. npm run start:headless -- --config <your-config.json>.
  4. curl the proxy port and read the stdout trace lines.
