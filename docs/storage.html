Storage

TL;DR Storage is the recording backend used by the recorder and mock plugins. Two flavours: fs (one JSON file per recording) or sqlite (single DB file). Both expose the same two-method API: save(rec) and findBest({method, path, query}).

Choosing a backend

"storage": { "type": "fs",     "path": "./recordings/httpbin" }
// or
"storage": { "type": "sqlite", "path": "./recordings/httpbin.db" }

If you omit storage, the proxy defaults to fs at ./recordings/<name>.

SQLite is optional better-sqlite3 is listed under optionalDependencies, so a fresh npm install may skip it. If a profile asks for type: "sqlite" and the package isn't installed, startup fails with: "SQLite storage requested but better-sqlite3 is not installed."

Recording shape

Every saved recording — whether on disk or in SQLite — looks like this:

{
  "id":      "1714492800123-ab12cd",
  "ts":      "2024-04-30T12:00:00.123Z",
  "method":  "GET",
  "path":    "/users/42",
  "query":   "lang=en",
  "url":     "/users/42?lang=en",
  "request": {
    "headers":      { "...": "..." },
    "body":         "...",            // utf-8 string OR base64
    "bodyEncoding": "utf8" | "base64"
  },
  "response": {
    "status":       200,
    "headers":      { "...": "..." },
    "body":         "...",
    "bodyEncoding": "utf8" | "base64"
  }
}

Body encoding

Bodies are stored as raw UTF-8 strings when they look "printable", otherwise base64. The check is simple: a body is treated as text if it contains no bytes in 0x00–0x08 or 0x0E–0x1F. See encodeBody() / decodeBody() in src/storage.js.

Filesystem backend

Class: FsStorage in src/storage.js.

  • Each recording becomes one file: <METHOD>__<path>__<id>.json, where the path is sanitized (non-alphanumeric → _).
  • save() writes synchronously with JSON.stringify(..., null, 2).
  • findBest() reads every file in the dir, filters by method + path. With a query string, only an exact query match counts. Without one, the newest match by ts wins.

SQLite backend

Class: SqliteStorage in src/storage.js.

Single table:

CREATE TABLE recordings (
  id      TEXT PRIMARY KEY,
  ts      TEXT NOT NULL,
  method  TEXT NOT NULL,
  path    TEXT NOT NULL,
  query   TEXT NOT NULL,
  url     TEXT NOT NULL,
  payload TEXT NOT NULL          -- the full JSON record
);
CREATE INDEX idx_recordings_lookup ON recordings (method, path, ts);

findBest() issues one indexed lookup ordered by ts DESC; the full record is decoded from the payload column.

Public API (for plugin authors)

// every plugin gets this on ctx.storage
storage.save(rec)                                 // returns id (sqlite) or path (fs)
storage.findBest({ method, path, query })         // returns a record or null

Helpers exported from src/storage.js:

buildRecord({ method, urlPath, query, req, res })   // constructs the shape above
decodeBody(rec.response | rec.request)              // returns Buffer

Used by the recorder plugin (to build records before saving) and the mock plugin (to decode replayed bodies).
