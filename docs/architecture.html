Architecture

TL;DR A request comes in → plugins run top-to-bottom (onRequest) → first plugin to set ctx.response wins → otherwise we forward to the upstream → all plugins run onResponse → we write the response back to the client.

Request lifecycle

This is the path every HTTP request takes through src/proxy.js:

   client                         proxy                      upstream
     │                              │                            │
     │ ─── HTTP request ──────────▶ │                            │
     │                              │                            │
     │                  ┌───────────┴──────────────┐             │
     │                  │ 1. readBody(req)         │             │
     │                  │ 2. build ctx object      │             │
     │                  │    {req, response:null}  │             │
     │                  └───────────┬──────────────┘             │
     │                              │                            │
     │                  ┌───────────┴──────────────┐             │
     │                  │ runOnRequest(plugins)    │             │
     │                  │ ────────────────────────  │             │
     │                  │  cors → latency → bucket  │             │
     │                  │   → mock → recorder → tap │             │
     │                  │                          │             │
     │                  │ stops at first plugin    │             │
     │                  │ that sets ctx.response   │             │
     │                  └───────────┬──────────────┘             │
     │                              │                            │
     │                  ┌───────────┴──────────────┐             │
     │                  │ ctx.response still null? │ ── yes ──▶  │
     │                  │ → forwardManual(req)     │             │
     │                  │   set ctx.meta.source    │ ◀── 200 ─── │
     │                  │   = 'proxy'              │             │
     │                  └───────────┬──────────────┘             │
     │                              │                            │
     │                  ┌───────────┴──────────────┐             │
     │                  │ runOnResponse(plugins)   │             │
     │                  │ (every plugin runs;      │             │
     │                  │  e.g. recorder saves,    │             │
     │                  │  tap emits TUI events,   │             │
     │                  │  cors adds headers)      │             │
     │                  └───────────┬──────────────┘             │
     │                              │                            │
     │ ◀──── HTTP response ─────────│                            │
     │                              │                            │
     │                              │ logger.trace('request')    │
Why two phases? onRequest is for short-circuiting (mock, bucket hit, CORS preflight, injected failure). onResponse is for observing or decorating the final response (recorder saves it, tap streams it to the TUI, cors stamps headers on it).

The ctx object

Every plugin gets the same ctx object. Mutate it freely; plugins later in the chain see your changes.

ctx = {
  config,    // the parsed profile JSON (read-only by convention)
  storage,   // FsStorage | SqliteStorage instance
  logger,    // shared EventEmitter logger

  req: {
    method,           // 'GET', 'POST', ...
    url,              // raw, includes query string
    path,             // url with query stripped
    query,            // string after '?', '' if none
    headers,          // node http headers object
    body,             // Buffer (already fully read)
  },

  response: null | {  // set by a plugin to short-circuit
    status,           // number
    headers,          // plain object
    body,             // Buffer
  },

  forwardUpstream,    // () => Promise<{ status, headers, body }> for shadow observers

  meta: {
    source: null | 'proxy' | 'mock' | 'ret_rec'
          | 'ret_rec_fallback' | 'bucket' | 'cors'
          | 'latency_fail',
    startedAt,        // Date.now() timestamp captured when handling began
  },
}

WebSockets

The HTTP server listens for upgrade events and hands them to src/ws-proxy.js. It performs the upstream WebSocket handshake, parses text/binary frames, calls runOnWsMessage(), then re-encodes frames for the correct peer. Plugins can set ctx.drop to suppress a frame, mutate ctx.frame.data, or push response frames into ctx.inject.

wsCtx = {
  config, storage, logger,
  ws: { connectionId, direction: 'client' | 'server', url, path, query, headers },
  frame: { opcode, binary, data: Buffer, text: string | null },
  drop: false,
  inject: [],
  meta: { source: 'proxy' | 'ws_tap_mock' | 'ws_tap_replay' | 'ws_tap_drop' }
}
